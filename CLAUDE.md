# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this package is

`packagepal` is a tiny R package whose entire user-facing surface is one function, `checklist(type)`, which prints a `cli`-formatted ordered list of steps for either building a new R package (`"package"`) or submitting one to CRAN (`"CRAN"`). There is no business logic beyond string dispatch — the value of the package lives in the *content* of the checklist items in `R/checklists.R`.

## Architecture

- `R/checklists.R` — all logic. `checklist()` is the only exported function and dispatches by string to two internal helpers, `package_checklist()` and `cran_checklist()`. Both helpers are a single `cli::cli_ol(c(...))` call. Invalid `type` values fall through to a `cli::cli_inform()` `"x"` message and return `invisible()` — no condition is raised, so callers cannot `tryCatch()` an invalid type.
- `R/packagepal-package.R` — package-level doc + the `usethis namespace` block where `@importFrom` directives live. Imports are split between *Imports* in `DESCRIPTION` (`cli`, `usethis`, `utils`) and explicit `@importFrom` here.
- `man/` is generated by roxygen2 — never hand-edit. Run `devtools::document()` after touching roxygen comments.
- `NAMESPACE` is also generated — do not hand-edit.

## Editing the checklists

Each item in `cli::cli_ol()` uses `cli` inline markup: `{.code ...}` for code, `{.pkg ...}` for package names, `{.href [text](url)}` for links, `{.emph ...}` for emphasis. Preserve this style when adding items. The CRAN list also embeds bare `{.href [...](...)}` links to chapters of *R Packages (2e)*.

## Testing — snapshot-based

The whole test suite is `expect_snapshot()` against the rendered `cli` output. Snapshots live in `tests/testthat/_snaps/checklists.md`.

**This means any text change to a checklist item will fail tests until you re-record the snapshot.** Workflow:

```r
devtools::test()                  # see which snapshots diverged
testthat::snapshot_review()       # interactively diff/accept (or reject)
# or, if you trust the change:
testthat::snapshot_accept()
```

Then commit the updated `tests/testthat/_snaps/checklists.md` alongside your `R/checklists.R` change. Skipping the snapshot update will break CI (R-CMD-check uploads snapshot diffs on failure — see `.github/workflows/R-CMD-check.yaml`).

## Common commands

Run from the package root in an R session (the project's `.Rprofile` auto-loads `devtools`, `testthat`, and `usethis` when interactive):

| Task | Command |
| --- | --- |
| Run all tests | `devtools::test()` |
| Run a single test file | `devtools::test_active_file("tests/testthat/test-checklists.R")` |
| Re-render docs after roxygen edits | `devtools::document()` |
| Full R CMD check (what CI runs) | `devtools::check()` |
| Build the README from `README.Rmd` | `devtools::build_readme()` |
| Test coverage | `devtools::test_coverage()` |
| Load the package interactively | `devtools::load_all()` |

## Dependency management

The project uses **renv**. `renv.lock` is the source of truth for dependency versions. After pulling changes that modify `renv.lock`, run `renv::restore()`. After adding/removing an import, run `renv::snapshot()` to update the lockfile, and remember to also update `Imports:` in `DESCRIPTION`.

## CI

Two GitHub Actions workflows run on push/PR to `main`/`master`:

- `.github/workflows/R-CMD-check.yaml` — full `R CMD check` on Ubuntu. Uploads snapshots on failure, which is the easiest way to debug snapshot mismatches.
- `.github/workflows/test-coverage.yaml` — codecov upload.

There is no lint step in CI.
